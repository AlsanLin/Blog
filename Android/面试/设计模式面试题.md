# 创建型
## 1. 单例模式：DCL(双重检查锁定)+volatile为什么需要volatile？静态内部类方式的优点，枚举单例的好处
关于单例模式，在 Java 和 Android 开发中，DCL（双重检查锁定）和静态内部类是最常用的两种写法。要彻底理解它们，核心在于理解**并发环境下的指令重排序**和**类加载机制**。
### 一、 DCL 中为什么必须加 `volatile`？
DCL（Double-Checked Locking）的典型代码如下：
```java
public class Singleton {
    private static volatile Singleton instance; // 必须加 volatile
    private Singleton() {}
    public static Singleton getInstance() {
        if (instance == null) { // 第一次检查
            synchronized (Singleton.class) {
                if (instance == null) { // 第二次检查
                    instance = new Singleton(); // 问题出在这一行
                }
            }
        }
        return instance;
    }
}
```
如果不加 `volatile`，在多线程环境下可能会出现**对象未完全初始化就被其他线程使用**的严重问题。原因如下：
#### 1. `new Singleton()` 并不是一个原子操作
在 JVM 底层，`instance = new Singleton();` 实际上会被编译成以下 3 条指令：
1. **分配内存空间**：给 `Singleton` 对象分配内存。
2. **初始化对象**：调用构造函数，初始化成员变量。
3. **引用指向内存**：将 `instance` 引用指向刚分配的内存地址。
#### 2. 指令重排序导致的 NPE
在现代编译器和 CPU 中，为了优化性能，可能会对指令进行**重排序**。指令 2 和指令 3 之间没有数据依赖关系，因此 JVM 可能会将执行顺序重排为 **1 -> 3 -> 2**。
**致命场景模拟：**
* **线程 A** 进入同步块，执行 `instance = new Singleton();`。发生了指令重排序，执行了 1（分配内存）和 3（将引用指向内存）。**此时 `instance` 已经不等于 `null` 了，但对象还没被初始化（构造函数没执行）。**
* **线程 B** 此时来到第一次检查 `if (instance == null)`，发现不为 null，**直接返回了未完全初始化的 `instance` 对象**。
* **线程 B** 使用该对象时，访问其成员变量，触发 `NullPointerException` 或其他异常。
#### 3. `volatile` 的作用
volatile 关键字在这里的作用是**禁止指令重排序**。它相当于加了一道内存屏障，保证 `new` 操作的三个指令严格按照 1 -> 2 -> 3 的顺序执行。只有当对象完全初始化（指令2执行完）后，引用才会被公开（指令3），从而保证了其他线程拿到的永远是完全初始化的对象。

### 二、 静态内部类方式的优点
静态内部类方式（也叫 Initialization-on-demand holder idiom）是利用 JVM 的类加载机制来实现延迟加载和线程安全的单例模式。
```java
public class Singleton {
    private Singleton() {}
    // 静态内部类
    private static class Holder {
        private static final Singleton INSTANCE = new Singleton();
    }
    public static Singleton getInstance() {
        return Holder.INSTANCE;
    }
}
```
**核心优点：**
1. **天然的线程安全（无需同步锁）**：
   JVM 在加载类的过程中，会执行类的 `<clinit>` 方法（静态变量和静态代码块的初始化）。JVM 会使用同步机制保证一个类的 `<clinit>` 方法在多线程环境中被正确地加锁、同步。如果多个线程同时尝试初始化 `Holder` 类，只有一个线程能执行，其他线程会被阻塞。
2. **完美的延迟加载（Lazy Initialization）**：
   只有当外部调用 `getInstance()` 方法时，`Holder` 内部类才会被主动使用，从而触发类加载和 `INSTANCE` 的实例化。如果不调用，对象永远不会创建，节约了内存。
3. **代码简洁，性能极高**：
   相比 DCL 的繁琐和显式加锁，静态内部类完全由 JVM 底层机制保证，没有任何同步代码块，执行效率极高。
---
### 三、 枚举单例的好处
Effective Java 作者 Joshua Bloch 极力推荐使用枚举来实现单例：
```java
public enum Singleton {
    INSTANCE;
    
    public void doSomething() { ... }
}
```
**核心好处：**
1. **最简洁的写法**：默认就是单例，不需要手写复杂的逻辑。
2. **绝对的线程安全**：枚举实例的创建本身就是线程安全的，由 JVM 底层保证。
3. **防反射攻击**：
   通过反射的 `newInstance()` 方法可以暴力破坏 DCL 和静态内部类的私有构造函数来创建新对象。但 Java 规范规定，**通过反射创建枚举对象时会直接抛出 `IllegalArgumentException` 异常**，从根本上杜绝了反射破坏单例。
4. **防反序列化破坏**：
   普通的 Java 类如果实现了 `Serializable`，反序列化时会通过反射调用构造器创建新对象，破坏单例（需要重写 `readResolve()` 方法来防止）。而枚举类在序列化和反序列化机制上有特殊处理，**JVM 保证每次序列化/反序列化返回的都是同一个实例对象**。
### 总结
* **DCL**：历史悠久，了解并发底层机制的最佳教材，但写法繁琐且依赖 `volatile`。
* **静态内部类**：日常 Java/Android 开发中最推荐的写法，兼顾了代码优雅、延迟加载和高性能（前提是单例不需要防反射和防反序列化）。
* **枚举单例**：最安全、最简洁的写法，能防御反射和反序列化攻击，但如果不涉及这些安全问题，有些人觉得它不够直观。


## 2. Kotlin的object声明的是哪种单例？和Java单例有什么区别
Kotlin的`object`声明的是**饿汉式单例**（在类加载到JVM时即初始化）。
与Java单例的主要区别：
1. **代码极简**：仅需`object Singleton`一行代码，Java需手写私有构造、静态实例及`getInstance()`方法。
2. **天生线程安全**：由编译器在底层保证初始化的线程安全，Java需手动同步（如双重检查锁）。
3. **非延迟加载**：不支持传参构造，加载即初始化；Java单例可灵活实现延迟加载或传参（如DCL模式）。

## 3. 建造者模式：AlertDialog.Builder为什么用建造者？链式调用和不可变对象的配合？
**为什么用建造者？**
1. **参数过多**：AlertDialog配置项极多（标题、按钮、主题等），构造器或set方法会导致参数列表过长或调用臃肿。
2. **隔离构造与表示**：将复杂的内部构建逻辑隐藏在Builder中，对调用方屏蔽了Dialog的具体组装过程，且便于扩展新配置。
**链式调用和不可变对象的配合？**
3. **链式调用（流式API）**：Builder的配置方法都返回`this`（即Builder自身），使代码可读性极高，一行代码即可完成复杂配置。
4. **不可变对象的配合**：在`create()`或`show()`被调用前，Builder本身是可变的配置容器。一旦调用创建方法，这些参数就被传递给目标Dialog并**“冻结”为不可变状态**。这种模式实现了“可变的构建过程”与“不可变的最终产物”的完美解耦，保证了对象的线程安全性和内部数据一致性。

## 4. 工厂方法vs抽象工厂vs简单工厂的区别，各举一个Android源码中的例子
**核心区别：**
1. **简单工厂**：一个工厂类，根据传入参数用`switch/if`创建不同产品。违反开闭原则。
2. **工厂方法**：一个工厂接口对应一个产品接口，每增加一个产品就新增一个具体工厂类。符合开闭原则。
3. **抽象工厂**：一个工厂接口创建**一族相关或互相依赖**的多个产品。应对产品维度（族）的扩展，不应对产品种类的增加。
**Android源码例子：**
4. **简单工厂**：`Fragment.instantiate(Context, className)`。传入类名字符串，内部通过反射直接生产出对应的Fragment实例。
5. **工厂方法**：`ListActivity`或`Activity.onCreatePanelMenu()`。系统定义了创建菜单的抽象流程（工厂方法），具体的`Activity`（具体工厂）决定实例化哪种具体的`MenuBuilder`（具体产品）。
6. **抽象工厂**：`javax.xml.parsers`包中的`SAXParserFactory`与`DocumentBuilderFactory`。根据不同的XML解析环境，工厂能同时创建出一整套相关的Parser对象和底层的Handler对象族。

## 5. 原型模式：Intent.clone()和Bundle.clone()的实现？深拷贝vs浅拷贝
**Intent.clone() 和 Bundle.clone() 的实现：**
两者均实现了`Cloneable`接口并重写了`clone()`方法。底层调用`super.clone()`（即`Object.clone()`）进行内存复制，随后将内部的集合（如`Bundle`内部的`ArrayMap`）也进行克隆，以防止修改副本影响到原件。
**深拷贝 vs 浅拷贝：**
1. **浅拷贝**：仅复制对象本身及基本类型字段的值，引用类型字段只复制“引用指针”。副本与原件共享同一个引用对象，修改其一会影响另一个。
2. **深拷贝**：递归复制对象本身及其所有引用指向的对象。副本与原件完全独立，互不影响。
**结论**：`Intent`和`Bundle`的`clone()`本质上是**浅拷贝**。它们只克隆了自身和第一层容器结构，但容器内存储的具体对象（如序列化对象、Parcelable对象）依然只复制了引用。


# 结构型
## 1. 适配器模式：RecyclerView.Adapter适配了什么？ListView的Adapter和RecyclerView的Adapter设计差异？
**RecyclerView.Adapter 适配了什么？**
适配了**数据集合**与**Item视图**。它将任意类型的数据源转换为UI能够渲染的视图项，并向RecyclerView屏蔽了具体数据结构，同时配合ViewHolder实现视图复用。
**ListView 与 RecyclerView 的Adapter设计差异：**
1. **ViewHolder内置与解耦**：ListView不强制使用ViewHolder（需开发者手动实现`getView()`并自行优化）；RecyclerView将ViewHolder作为泛型内置到Adapter体系，强制复用，开发者只需处理`onBindViewHolder()`。
2. **职责拆分**：ListView的Adapter把“创建视图”和“绑定数据”揉合在`getView()`中；RecyclerView拆分为`onCreateViewHolder()`（专门创建视图）和`onBindViewHolder()`（专门绑定数据），职责更单一。
3. **布局解耦**：ListView的Adapter自身承担了少量布局排列职责；RecyclerView彻底移除了布局逻辑，交由独立的`LayoutManager`处理，Adapter只负责“造数据项”，彻底解耦。

## 2. 装饰者模式：ContextWrapper的继承链条？一个Activity背后有几个Context？
**ContextWrapper的继承链条：**
`Context` (抽象接口) 
  -> `ContextWrapper` (标准装饰者，内部持有`mBase`引用) 
    -> `ContextThemeWrapper` (带主题的装饰者) 
      -> `Activity` (具体被装饰的组件)。
**一个Activity背后有几个Context？**
在内存中实际存在的Context对象通常是**1个**（Activity本身）。
**简要解释：**
虽然继承链条多层包装，但`ContextWrapper`并未`new`出新的底层Context，而是通过`attachBaseContext()`将应用的`ContextImpl`实例作为`mBase`注入。因此，Activity的Context操作最终全部委托给这同一个`ContextImpl`对象执行，这是典型的**装饰者模式**（动态增加/覆盖功能，但不改变底层核心对象）。

## 3. 代理模式：Binder的Proxy/Stub是哪种代理？静态代理和动态代理的区别？
**Binder的Proxy/Stub是哪种代理？**
是**远程代理**。
Stub（存根）运行在服务端，Proxy（代理）运行在客户端。客户端调用Proxy方法，Proxy通过Binder驱动将请求跨进程转发给Stub执行，对客户端而言“就像调用本地方法一样”。
**静态代理和动态代理的区别：**
1. **静态代理**：代理类在**编译期**就已确定，需开发者手动编写。代理类与被代理类实现同一接口，内部显式持有被代理类引用并转发调用。
2. **动态代理**：代理类在**运行期**动态生成，无需手动编写。通常利用反射机制（如Java的`Proxy.newProxyInstance`或Android的`IInterface`+`asInterface`）在运行时动态生成代理对象并拦截方法调用。

## 4. 外观模式：Retrofit/Glide的对外API为什么算外观模式？
**为什么算外观模式？**
1. **屏蔽底层复杂子系统**：Retrofit底层涉及OkHttp、ConverterFactory、CallAdapter、注解解析等；Glide底层涉及BitmapPool、Engine、DecodeJob、网络层等。子系统极其复杂。
2. **提供极简统一入口**：它们对外仅暴露一个优雅的接口（如`Retrofit.create()`或`Glide.with().load().into()`），将内部复杂的构建、调度、缓存逻辑全部隐藏，让调用者以极简的API完成复杂的网络请求或图片加载。

## 5. 桥接模式：Window和WindowManager的关系？为什么说这是桥接模式
**Window与WindowManager的关系：**
`Window`是抽象窗口接口（定义视图容器能力），`WindowManager`是窗口管理接口（负责添加、更新、移除View）。两者通过`Window.setWindowManager()`互相持有引用，紧密协作。
**为什么说这是桥接模式？**
1. **多维度的独立变化**：系统需要在两个独立维度上变化——“窗口的层级/类型”（如Application层、Activity层、系统悬浮窗等）和“窗口的管理/显示机制”（具体实现类如`WindowManagerImpl`）。
2. **桥接解耦**：如果不用桥接，需为每种Window和每种Manager组合生成大量子类（类爆炸）。桥接模式将二者拆分为两个独立的继承体系，并在`Window`内部持有一个`WindowManager`的引用作为“桥梁”。这样，Window的抽象逻辑和WindowManager的实现可以各自独立扩展，互不影响。


# 行为型
## 1. 观察者模式：LiveData和Flow各自的观察者实现有什么不同？LiveData.observe的生命周期感知是怎么实现的
**LiveData和Flow观察者实现的不同：**
1. **LiveData**：基于经典的观察者模式，**强依赖Lifecycle组件**。通常只能在主线程更新数据（`setValue`），通过`postValue`切回主线程。
2. **Flow**：基于协程的**冷流**模型，观察者需处于协程作用域中通过`collect`挂起收集。本身不感知生命周期，需借助`repeatOnLifecycle`等辅助函数才能安全地在生命周期内收集。支持背压和任意线程切换。
**LiveData.observe生命周期感知的实现：**
`observe`传入的`LifecycleOwner`和`Observer`会被包装成`LifecycleBoundObserver`。该包装类实现了`GenericLifecycleObserver`接口并注册到`LifecycleRegistry`中。当宿主的生命周期变化时，回调`onStateChanged`；如果是`DESTROYED`状态，LiveData会自动移除该Observer；如果是`STARTED`状态，则将最新数据分发给Observer。

## 2. 责任链模式：OkHttp拦截器链的调用流程？intercepter.proceed()做了什么？
**OkHttp拦截器链的调用流程：**
请求按顺序依次经过：自定义拦截器 → RetryAndFollowUp（重试）→ Bridge（补全头）→ Cache（缓存）→ Connect（建立连接）→ 自定义网络拦截器 → CallServer（真正读写I/O）。形成一条从上到下、再从下到上返回响应的U型链。
**interceptor.proceed()做了什么？**
1. **推进链条**：内部创建下一级拦截器对应的`RealInterceptorChain`（索引+1）。
2. **递归调用**：调用下一个拦截器的`intercept()`方法，将控制权交予下一层。
3. **返回处理**：等待下一层执行完毕并返回`Response`后，拿回控制权，对响应进行逆向处理（如解压、日志记录）并返回给上一层。

## 3. 策略模式：支付方式切换为什么是策略模式？和if-else的本质区别
**支付方式切换为什么是策略模式？**
因为不同的支付方式（微信、支付宝、银行卡）代表了完成“支付”这个目的的不同**算法或行为**。将每种支付方式独立封装成实现了同一接口的策略类，客户端持有一个策略引用，在运行时动态注入具体策略来执行支付。这完美符合策略模式“定义一系列算法，将每一个封装起来，并使它们可以互相替换”的定义。
**和if-else的本质区别：**
1. **开闭原则（扩展性）**：if-else将所有逻辑硬编码在一个方法内，新增支付方式必须修改源码（违背开闭原则）；策略模式只需新增策略类并注入，无需修改已有代码（符合开闭原则）。
2. **职责分离（内聚性）**：if-else把所有复杂逻辑揉在一个类中，代码随业务膨胀，极难维护；策略模式把每种算法的细节隔离到独立的策略类中，职责单一，互不干扰。
3. **消除条件分支**：if-else充斥着大量条件判断语句，运行时动态匹配；策略模式利用多态特性，在运行前已确定具体策略，消除了复杂的条件判断代码。

## 4. 状态模式：订单状态机为什么用状态模式而不是一堆if
**订单状态机为什么用状态模式而不是一堆if？**
1. **消除庞大的条件分支**：订单流转（如待支付→已支付→已发货→已完成）涉及大量状态判断。如果用if/else或switch，每个状态的处理逻辑全揉在一个方法里，随着状态增加，代码会变成极难维护的“巨型函数”。
2. **局部化状态行为**：状态模式将每个状态的行为封装到独立的类中（如`PaidState`处理发货，`ShippedState`处理确认收货）。修改某个状态的逻辑只需改对应类，完全不影响其他状态，符合**单一职责原则**。
3. **消除重复的状态校验**：在if/else中，每次行为发生前都要写满`if(order.status == XX)`的防御性判断；状态模式通过多态动态绑定当前状态，上下文无需关心当前是什么状态，直接委托给状态对象执行即可。
4. **符合开闭原则**：未来新增状态（如“退款中”），只需新增状态类并配置流转关系，无需修改原有的状态类或庞大的if-else分支。

## 5. 命令模式：Runnable/kotlin.Unit lambda是命令模式的体现，为什么
**为什么说Runnable和Kotlin Lambda体现了命令模式？**
命令模式的核心思想是：**将请求（动作）封装成一个对象**，使得你可以用不同的请求对客户端进行参数化、排队请求、记录日志，或撤销操作。
**1. Runnable 是命令模式的体现：**
*   **封装请求**：`Runnable` 接口将一段执行逻辑（动作）封装成了标准的对象。它的 `run()` 方法就等同于命令模式中的 `execute()` 方法。
*   **请求发送者与接收者解耦**：像 `Thread` 或线程池 `ExecutorService`（调用者/Invoker）只负责接收 `Runnable` 对象并调用其 `run()` 方法，它完全不需要知道这个任务内部具体要干什么、操作了哪些数据（接收者/Receiver）。
*   **参数化与排队**：因为任务被封装成了对象，所以你可以把多个 `Runnable` 放入队列中（如线程池的阻塞队列）延迟执行、定时执行，甚至序列化传输到另一个节点执行。
**2. Kotlin Lambda（Unit返回类型）是命令模式的体现：**
*   Kotlin 中的 `() -> Unit` 类型 Lambda 表达式在编译后，本质上会被编译为一个实现了 `Function0<Unit>` 接口的匿名类对象。
*   这与 `Runnable` 的底层机制完全一致：将一段行为代码封装成了一个实现了特定接口的对象。通过传递这个 Lambda，你实际上是在传递一个“命令对象”。
*   在 Android 开发中，如 `view.setOnClickListener { /* 跳转 */ }`，传入的 Lambda 就是一个被封装的命令对象，它被持有在 View 的监听器中，等待点击事件触发时被“执行”。
**总结**：只要是将“行为”或“动作”参数化，包成了一个对象传来传去并延迟调用，就是命令模式的应用。Runnable 和 Lambda 正是语言层面提供的最轻量级的命令封装工具。

## 6. 中介者模式：MVP中的Presenter作为View和Model的中介者？和MVC中Controller的差异
**MVP中的Presenter作为中介者：**
在MVP中，View和Model被完全解耦，互不相识。Presenter扮演着中介者的角色：View通过接口将用户事件委托给Presenter，Presenter处理逻辑后调用Model获取数据，然后再通过View的接口将数据回传给View展示。所有跨对象的通信都必须经过Presenter路由。
**和MVC中Controller的差异：**
1. **连接紧密程度（控制力）**：MVC中Controller通常较“薄”，只负责将请求分发给对应的Model和View，View和Model往往可以直接交互；MVP中Presenter较“厚”，不仅路由，还直接持有View的接口引用，主导View的刷新，View和Model绝对隔离。
2. **对View的依赖方式**：MVC中Controller往往直接依赖具体的View（如Android的Activity本身既是Controller又是View）；MVP中Presenter依赖的是抽象的View接口，使得Presenter可以进行纯粹的单元测试，彻底解除了对系统UI框架的依赖。
3. **核心角色定位**：MVC的Controller更像是“请求转发器”；MVP的Presenter是真正的“业务逻辑大管家+通信中介”。
